## DebtFlow – омниканальный конструктор коммуникаций (MVP)

DebtFlow — это прототип веб‑сервиса для работы с задолженностью, который показывает, как можно:

- загружать списки должников;
- настраивать шаблоны сообщений (SMS / Email / Call);
- запускать кампании по выбранным сегментам;
- имитировать отправку по каналам (mock‑провайдеры);
- анализировать результаты на дашборде (объёмы, успешность, стоимость).

Проект создан как демонстрационный MVP для портфолио: он сочетает юридический контекст (152‑ФЗ, взыскание задолженности), операционный опыт (колл‑центры, лидогенерация) и современный технический стек.

### Экспертиза автора (синергия трёх ролей)

DebtFlow отражает комбинацию трёх областей:

- **Юрист по взысканию** — понимание требований к обработке ПДн (152‑ФЗ), отраслевых ограничений и типовых сценариев коммуникации с должниками.
- **Операционный директор** — опыт в лидогенерации для банков и управлении процессами колл‑центров: метрики (объём, успешность, стоимость), сегментация, многоканальность (SMS, email, звонки).
- **Технический специалист** — реализация продукта: SQL/аналитика, API, автоматизация (в т.ч. голосовые боты), современный стек (Next.js, Supabase, TypeScript).

В одном продукте соединены юридическая грамотность, операционное видение и умение доводить идею до работающего кода — это и демонстрирует проект.

## Основные страницы

- `/dashboard` — дашборд: количество сообщений, успешность, стоимость, график по дням, недавние коммуникации.
- `/debtors` — список должников, импорт CSV, фильтры.
- `/templates` — шаблоны сообщений (SMS / Email / Call) с редактором и счётчиком символов для SMS.
- `/channels` — настройки mock‑каналов и их стоимости.
- `/communications/new` — мастер запуска кампании (3 шага).
- `/communications` — лог отправок.
- `/scorer` — Pre-Collection Scorer: скоринг должников (модель на синтетических данных), сегменты и рекомендуемый канал.
- `/compliance` — соответствие 230‑ФЗ: лимиты контактов (звонки/SMS), время, экспорт для отчётности ФССП.
- `/admin/seed` — административная страница для заполнения демо‑данными (защищена `SEED_SECRET`).
- `/legal/privacy`, `/legal/terms` — заглушки юридических документов.

## Ограничения демо и модель данных

Этот репозиторий — **портфолио‑MVP**, а не готовый к продакшену изолированный SaaS. Ниже — как устроено намеренно и что иметь в виду при ревью кода.

- **Одна демо‑организация.** Данные в интерфейсе привязаны к фиксированному идентификатору организации в коде. Все зарегистрированные пользователи демо фактически работают с **общей песочницей**, а не с отдельными несмешиваемыми организациями.
- **Вход (Supabase Auth) и таблица `public.users`.** Регистрация создаёт пользователя в Supabase Auth, но **не создаёт автоматически** строку профиля в `public.users` с `organization_id`. Файл `supabase/rls.sql` рассчитан на связку `auth.uid()` ↔ строка в `public.users`. Если включить RLS как в репозитории, без донастройки (триггер на `auth.users`, серверный провижинг профиля и т.п.) клиентские запросы к данным **могут не проходить** политики. На этапе демо часто проще **не применять** `rls.sql` или явно довести провижинг профиля до конца — иначе слова «RLS в проекте» и фактическое поведение расходятся.
- **Серверные API (импорт, compliance, scorer).** Маршруты под демо не реализуют полноценную продовую модель «пользователь может видеть только свою организацию»: например, импорт должников не привязан к сессии, а в query допускается подстановка `organizationId`. Пока в базе одна организация, риск ограничен; при настоящей многотенантности потребуется **авторизация на сервере** и вывод `organization_id` из доверенного контекста (JWT / service role), а не из произвольного параметра запроса.
- **Seed (`/api/admin/seed`).** Доступ ограничен заголовком `SEED_SECRET`, но запись в БД идёт через серверный клиент с **anon**‑ключом. При строгом RLS типичный продовый вариант — выполнять такие операции с **service role** только на сервере (ключ не в `NEXT_PUBLIC_*`).

Итог: **README описывает целевую архитектуру** (PostgreSQL, задел под RLS, Auth), а текущий код оптимизирован под **быстрый показ сценария** в одной демо‑организации. Для собеседования или тендера полезно опираться на этот раздел и на блок «Дальнейшее развитие».

## Запуск локально

1. Установите зависимости:

```bash
npm install
```

2. Создайте в корне файл `.env.local` на основе `.env.example` и заполните:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SEED_SECRET` (любой сложный секрет, тот же нужно будет указать в Vercel).

3. В Supabase:

- создайте новый проект;
- выполните SQL из файлов:
  - `supabase/schema.sql` (таблицы и связи),
  - при необходимости `supabase/migrations/add_call_to_templates.sql` (если проект создан до добавления канала Call),
  - `supabase/rls.sql` (политики RLS — **опционально**; см. раздел <a href="#demo-limits">«Ограничения демо и модель данных»</a>: при включении RLS без провижинга `public.users` приложение может не получать данные).

4. Запустите dev‑сервер:

```bash
npm run dev
```

Откройте `http://localhost:3000`:

- сначала зарегистрируйтесь/войдите;
- перейдите на `/admin/seed` и выполните заполнение демо‑данными;
- после этого будут доступны живые данные на `/debtors`, `/templates`, `/dashboard` и в логах.

## Деплой на Vercel

1. Залейте репозиторий на GitHub.
2. В Vercel:
   - создайте новый проект из этого репозитория;
   - в настройках проекта добавьте переменные окружения:
     - `NEXT_PUBLIC_SUPABASE_URL`
     - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
     - `SEED_SECRET`
3. В **проекте Supabase, который используется для продакшена** (его URL и anon key указаны в Vercel), откройте **SQL Editor** и выполните скрипт из файла `supabase/migrations/add_call_to_templates.sql`. Без этой миграции шаблон с каналом Call не будет создаваться при seed.
4. Дождитесь деплоя и откройте выданный домен вида `https://debtflow-*.vercel.app`.
5. На проде выполните:
   - регистрацию пользователя,
   - запуск `/admin/seed`,
   - прохождение сценария: Scorer → Debtors → Templates → Communications → Dashboard → Compliance.

## Юридические заметки

- В демо‑версии используются исключительно синтетические данные, не являющиеся персональными.
- В реальном продукте должны быть учтены требования 152‑ФЗ:
  - хранение и обработка ПДн на территории РФ;
  - получение согласий субъектов;
  - регламенты доступа и аудита.
- При взаимодействии с должниками по потребительским кредитам/займам необходимо соблюдать требования 230‑ФЗ «О потребительском кредите (займе)»: время и частота контактов, обязательная информация в сообщениях, ограничения на способы взаимодействия.

На страницах `/legal/privacy` и `/legal/terms` размещены базовые заглушки под политику конфиденциальности и пользовательское соглашение.
